# Figma Prototype → MCP → Admin 구현 가이드

## 목적과 범위

Figma 화면과 프로토타입은 Admin React 구현의 출발점이다. MCP로 가져온 결과는 화면 구조·디자인 규칙·인터랙션 의도이며, 실제 데이터·권한·저장 동작까지 자동으로 완성하지는 않는다.

적용 화면: `SCR-009`~`SCR-011`, `SCR-015`~`SCR-019`.

## 어디까지 가져오고, 무엇을 구현하는가

| 구분 | Figma/MCP로 반영할 것 | 코드에서 구현할 것 |
| --- | --- | --- |
| UI | 헤더, 사이드바, 표, 필터, 폼, 모달, 카드의 배치와 스타일 | 페이지/재사용 컴포넌트 분리 및 반응형 보정 |
| 프로토타입 | 화면 이동, 모달, 탭·필터 변화의 흐름 | 라우팅, 실제 이벤트 핸들러, 뒤로가기 정책 |
| 데이터 표시 | 주문·메뉴·매출의 더미값 | `API-007`~`API-015`, loading/empty/error, 값 포맷팅 |
| 변경 동작 | 버튼 및 성공/실패 화면의 외형 | 인증·권한, 검증, 중복 실행 방지, 409·5xx 복구 |
| 품질 | focus/disabled 등 상태의 디자인 | 키보드 접근성, aria-label, lint/build, 화면 테스트 |

## 화면별 작업 연결

| 화면 | 디자인에서 확정할 것 | 구현에서 연결할 것 |
| --- | --- | --- |
| `SCR-009` 주문 목록 | 필터, 테이블, 상태 배지, 빈 결과 | `API-007`, 조회 조건·페이지네이션·loading/error |
| `SCR-010` 주문 상세 | 상세 정보, 상태 변경 모달 | `API-008`, 중복 클릭 차단, 409 시 최신 상태 재조회 |
| `SCR-011` 품절 관리 | 목록, 토글, 사유 표현 | `API-009/010`, 실패 시 이전 값 복구 |
| `SCR-016/017` 메뉴 관리 | 검색, 목록, 등록/수정 폼 | `API-011/012`, 필수값·가격 검증 및 입력값 보존 |
| `SCR-018` 결제수단 | 활성 상태와 정렬 UI | `API-013/014`, 저장 실패 복구 |
| `SCR-019` 매출 | 기간 필터, 요약 카드, 차트/표 | `API-015`, 날짜 검증 및 empty/error 분리 |

## 권장 순서

1. Figma에 기본·hover·focus·disabled·selected·loading·empty·error 상태와 모달을 준비한다.
2. 반복 요소(버튼, 상태 배지, 필터, 다이얼로그, 테이블 셀)를 컴포넌트화한다.
3. MCP 결과를 페이지 골격과 스타일 기준으로 적용한다.
4. `AdminApp.jsx` 라우팅, API 모듈, store를 연결한다.
5. 성공·실패·충돌·재시도 동작을 구현한다.
6. `TC-A01`~`TC-A06`, 접근성, `npm.cmd run lint`, `npm.cmd run build`를 확인한다.

## CSS 기준

```text
src/
  styles/                 # reset, 폰트, 색상/간격 토큰, 앱 공통 규칙
  pages/admin/            # 화면 JSX와 화면 전용 스타일
  components/             # 재사용 UI와 해당 컴포넌트 스타일
```

- 전역에는 reset, 폰트, CSS 변수, 앱 공통 레이아웃만 둔다.
- 주문 목록의 필터·테이블처럼 한 화면 전용 규칙은 해당 페이지에 둔다.
- 여러 화면에서 반복되는 버튼, 배지, 모달, 입력 필드만 공통 컴포넌트로 추출한다.
- 클래스명은 범위를 드러낸다. 예: `.admin-order-list__filter`, `.admin-sales__summary`.

## Figma 전달 전 체크리스트

- [ ] 각 프레임에 `SCR` ID와 목적이 있다.
- [ ] 기본·hover·focus·disabled·selected·loading·empty·error 상태가 있다.
- [ ] 모달의 열기·닫기·취소·확인 흐름이 프로토타입에 연결됐다.
- [ ] API 값과 디자인용 더미값을 구분했다.
- [ ] 401/403/409/5xx 및 저장 실패 안내를 설계했다.

## 완료 기준

디자인 외형과 주요 흐름이 일치하고, API 성공·실패·빈 결과·권한/충돌 상태에서 문서화된 동작을 수행해야 완료다. 프로토타입의 클릭 연결만으로는 완료가 아니다.
